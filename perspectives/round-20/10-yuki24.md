# Round 20 field notes — Yuki Nishijima answers the 404

*Built: `examples/helpful_404.rb` — did-you-mean as a product
feature at the three doors where users arrive with typos: URLs,
config keys, CLI commands. One engine, three renderers, and the
conservatism rule proven: strangers get honest silence.*

## What I built and why

Round 14 I threaded suggestions into the framework's errors. This
round's brief — apply the gem to problems outside itself — is the
argument I've been making since did_you_mean shipped: **this was
never a Ruby feature. It's a product feature that happens to be
easy in Ruby.** Every system has three doors where humans arrive
with typos, and at all three, the system is holding the complete
answer key at the exact moment it says "not found":

```
router  /userz/42    -> 404 - did you mean /users/:id?
router  /xyzzy       -> 404                        (a stranger; silence)
config  max_retrys   -> did you mean 'max_retries'?
cli     stauts       -> did you mean 'status'?
cli     shipit       -> no command 'shipit'        (aspirational; silence)
score: 3 typos rescued, 3 strangers given honest silence
```

The engine is the framework's own `Suggestions` module — the same
Levenshtein that fixes contract keys fixes URLs and subcommands,
because a typo doesn't care what layer it lives in. Each door only
supplies two small things: a normalizer (the router compares path
*segments*, not whole paths) and a renderer (a 404 body, a boot
error, a CLI message). Three doors, maybe twelve lines each.

## The half of the feature everyone skips

The rescued typos are the demo; the **silences are the product**.
`shipit` is not close to any command; `databese_url` is not close
to any key we actually have; `/xyzzy` is nobody's route. All three
get plain not-found — no guess — because the threshold scales with
word length and refuses to stretch. A wrong suggestion ships a
confused user somewhere *confidently*, which is strictly worse than
a dead end: the dead end at least tells the truth. Half the trials
in this example test the silence, and I'd argue they're the
important half. Suggestion features die in production not because
they miss typos but because they once suggested something absurd
and lost the user's trust forever.

## Notes

- The router door demonstrates the one non-obvious integration
  decision: normalize before comparing. Suggesting against raw
  paths would make `/userz/42` compete with `/users/:id` as whole
  strings — the `:id` noise wrecks the distance. Compare the
  segment a human actually typed wrong.
- Where else the answer key is already in hand: environment
  variable names, GraphQL fields, feature-flag keys, kwargs in
  public APIs, header names. Every "unknown X" error in your
  codebase is this example with a different renderer.
- Cost check, since someone always asks: the candidate lists here
  are tens of items and the pass is microseconds. If your route
  table has ten thousand entries, suggest against a prefix-filtered
  subset. The feature scales down gracefully and up honestly.

## Verdict

Three doors, three rescues, three honest silences, one engine
borrowed from the framework's own toolbox. The error message is a
UI, the answer key is already in the system's hand at the moment of
failure, and the conservatism threshold is what keeps the feature
trustworthy enough to keep. Spend the Levenshtein.
